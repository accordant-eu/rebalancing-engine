## Tranche 5/6 Execution: RBAC Implementation & API Hardening

### Files Touched

**Backend:**
- `src/api/server.ts`
  - Added `isSuperadmin` boolean to the JWT payload decoding for easier assertions across endpoints.
  - Implemented `requireAdmin` and `forbidViewer` middlewares to secure mutation endpoints.
  - Fixed a severe bug where `tenantId === 'superadmin'` was incorrectly hardcoded across all endpoints, replacing it with correct `req.isSuperadmin` resolution.
  - Fixed a bug in `PUT /api/models/:id` where Tenant Admins could overwrite models owned by other tenants by explicitly verifying ownership before updating.

**Frontend:**
- `web/src/App.tsx`
  - Updated login flow to set the `userRole` to `Superadmin` using the new `isSuperadmin` boolean from the API response.
  - Wrapped sensitive UI elements (like "Create New Model Mandate") in authorization guards (`userRole === 'Admin' || userRole === 'Superadmin'`).

### Decisions
1. **Middleware Over Explicit Checks:** Chose to abstract RBAC role checks into `requireAdmin` and `forbidViewer` middlewares to keep endpoint handlers clean and reduce boilerplate.
2. **JWT Payload Expansion:** Added `isSuperadmin` to the JWT/API key request object payload. This standardizes Superadmin detection instead of relying on string equality against the `SUPERADMIN_TENANT_ID` env variable in every route.
3. **Implicit Tenant Scoping:** Kept `requireAdmin` scoped only to the `Admin` role without differentiating Superadmin, relying on the fact that models and actions are already securely scoped to `req.tenantId` during creation.

### Open Questions / Known Issues
- None at this time. All RBAC tests passed successfully.
