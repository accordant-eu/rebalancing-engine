# Engine Architecture and Domain Model

Date: 2026-06-14

## Design Principles

The core calculation engine adheres to a strict set of architectural principles, ensuring it remains resilient, scalable, and adaptable as a stateless module within the broader Live Agent orchestrator.

1. **Deterministic Calculation Core:** The engine must function as a pure mathematical core. Given the identical portfolio state, target allocation, and market prices, the engine must produce the exact same trade proposal every single time.
2. **Strategy Modules over Hard-Coded Logic:** The system utilizes the **Strategy** design pattern. This defines a family of rebalancing algorithms (e.g., threshold, calendar, manual) and encapsulates each one within a distinct module, separating common drift detection math from strategy-specific execution rules.
3. **Clear Separation from Execution:** The engine is responsible exclusively for portfolio analysis and trade proposal generation. It utilizes the **Command** pattern to encapsulate trade recommendations as independent instruction objects. Actual execution is delegated to the orchestrator layer.
4. **Policy-as-Configuration:** Rebalancing behaviors must be abstracted from the execution code into parameterized configurations.
5. **Standardized Data Lineage:** To ensure interoperability, the system's internal object structures align with the Banking Industry Architecture Network (BIAN) Service Landscape where applicable.

## Component Architecture

The calculation core is a stateless functional engine designed to be wrapped by a live orchestrator.

### Common Functional Building Blocks
The architecture aggressively extracts common functions utilized across all methodologies:
- `calculateMarketValues`: Applies price snapshots to holding quantities.
- `calculateCurrentWeights`: Determines the exact percentage composition.
- `calculateDrift`: Computes absolute and relative delta against targets.
- `routeCashInflows`: Allocates free cash to underweight assets to minimize selling friction.
- `applyMinimumConstraints`: Filters out economically unviable fractional trades based on policy.
- `generateAuditRecord`: Serializes the complete decision tree to a schema-constrained JSON object.

### Strategy-Specific Modules
The engine isolates variable algorithmic logic into swappable Strategy Modules:
- **Threshold/Tolerance-Band Module:** Evaluates drift against configured absolute and relative bands.
- **Calendar-Based Module:** Triggers based strictly on scheduled intervals, frequently used for systematic factor momentum strategies.
- **Manual Module:** Triggers an immediate full-reset rebalance based on an explicit command.

## Domain Model

| Entity                | Purpose                                                      | Key Fields                                             |
| :-------------------- | :----------------------------------------------------------- | :----------------------------------------------------- |
| **Account**           | The root entity representing the investor's sub-ledger.      | accountId, investorId, policyId, cashBalance           |
| **Instrument**        | The tradable security or asset class definition.             | instrumentId, symbol, assetClass, status               |
| **ModelPortfolio**    | A versioned collection of target weights.                    | modelId, version, allocations                          |
| **RebalancingPolicy** | The configuration defining active strategies and thresholds. | policyId, methodType, thresholds, minTradeSize         |
| **PortfolioState**    | A point-in-time snapshot of the account's economic reality.  | valuationTimestamp, totalValue, holdings               |
| **PriceSnapshot**     | The reference market data utilized for valuation.            | timestamp, instrumentId, price, asOf                   |
| **DriftMeasurement**  | The calculated delta between current and target weights.     | instrumentId, absoluteDrift, relativeDrift, status     |
| **TradeProposal**     | The overarching recommendation generated by the engine.      | proposalId, triggerReason, proposedTrades, turnover    |
| **ProposedTrade**     | The individual execution instruction within a proposal.      | instrumentId, action (BUY/SELL), quantity, value       |
| **AuditRecord**       | The immutable log of the entire decision-making process.     | eventId, timestamp, inputState, policySnapshot, output |

## API / Interface Sketch

The API design utilizes clear boundaries and structured JSON payloads.

**evaluateRebalance**
*Purpose:* Synchronously evaluates an account against its active policy and returns a comprehensive trade proposal. In the live agent context, this is called continuously or on state-change triggers by the orchestrator.

*Request Payload Example:*
```json
{  
 "accountId": "ACC-10001",  
 "simulateOnly": false,  
 "forceTrigger": false  
}
```

*Response Payload Example:*
```json
{  
 "rebalanceNeeded": true,  
 "triggerReason": "AAPL absolute weight 0.142 exceeds upper bound 0.150",  
 "proposalId": "PROP-9921",  
 "proposedTrades": [],  
 "simulation": { "estimatedTurnover": 0.04, "postTradeCash": 800.00 }  
}
```
