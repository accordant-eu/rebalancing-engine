# Use JSONL for persistent audit trails in MVP

## Context

In Tranche 4 (Production Hardening), the Live Agent needs a persistent sink for the immutable `AuditRecord` logs generated during every trade proposal. Previously, databases like SQLite or Postgres were considered. The `AuditRecord` generated by the core engine is a highly nested object containing arrays of trades, cash flows, and exact policy configurations.

## Options considered

1.  **SQLite/Postgres Database:** Normalizing the `AuditRecord` requires 10+ relational tables (audits, proposals, trades, holdings, snapshots). Alternatively, the entire JSON payload could be dumped into a single `TEXT` or `JSONB` column.
2.  **JSONL (JSON Lines) Flat File:** Appending each `AuditRecord` as a stringified single line to a `.jsonl` file.

## Decision

We will use JSON Lines (`.jsonl`) for the MVP persistent audit trail.

## Rationale

The primary requirement for Tranche 4 is an append-only, immutable regulatory log that proves exactly why the agent traded. A JSONL file natively provides append-only capabilities with zero setup, zero dependencies, and zero schema migration overhead. It can be easily queried from the command line using tools like `jq`. Since there is no immediate requirement for complex relational querying or API exposure, an embedded database introduces unnecessary complexity. If relational querying becomes necessary later, the `.jsonl` file can be deterministically parsed and bulk-loaded into a SQL database.

## Implementation impact

- Create an `AuditStorageAdapter` interface.
- Implement `FileAuditStorage` that uses Node's `fs.appendFile` to write to `data/audit-trail.jsonl`.

## Validation

- Verified that JSONL files can be appended to continuously without memory bloat.
- Verified that the logs can be easily tail-ed or parsed via `jq`.
