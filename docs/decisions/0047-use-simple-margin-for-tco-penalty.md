# Use simple margin constraint for TCO penalty

## Context

In Tranche 6 (Friction Optimization), the Live Agent was updated to consider the Total Cost of Ownership (TCO) of its proposed trades. A mathematical engine will propose a $10 trade to fix a 0.001% drift, but if the broker charges a $1 fixed fee, that trade incurs a 10% instant loss. We needed a Penalty Function to suppress trades where "Cost > Benefit".

## Options considered

1. **Option A (Simple Margin):** Configure a `maxFrictionBps` policy limit. If the estimated cost of the trade exceeds that percentage of the trade's total monetary value (e.g., > 0.50%), the trade is suppressed.
2. **Option B (Alpha Benefit):** Attempt to project the theoretical yield or expected return generated by moving the asset back to target, and suppress the trade if the friction cost exceeds that absolute projected alpha.

## Decision

We will use **Option A (Simple Margin)** for the MVP.

## Rationale

Option A acts as a dynamic "minimum trade size." If a $1 fee is applied to a $1,000 trade, the cost is 10 bps (accepted if limit is 50 bps). If applied to a $100 trade, the cost is 100 bps (suppressed). This is completely deterministic and requires no predictive modeling or assumptions about asset returns. Option B is deferred as it violates the deterministic, stateless mathematical philosophy of the core evaluation engine.

## Implementation impact

- Added `FrictionModel`, `FixedFeeModel`, and `PercentageSlippageModel` to `src/core/friction.ts`.
- Added `maxFrictionBps` to `RebalancingPolicy`.
- Updated `generateTradeProposal` to conditionally apply `applyFrictionPenalties`, filtering out trades that violate the limit and appending a `FRICTION_COST_EXCEEDED` warning.
- Updated the Command Center Dashboard to explicitly render friction warnings.

## Validation

- Verified via unit tests that fixed fee models successfully suppress small trades while allowing large trades to pass under the same bps limit.
